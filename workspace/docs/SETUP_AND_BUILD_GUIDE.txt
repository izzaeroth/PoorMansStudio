Windows Setup and Build Guide

This section explains how to set up a Windows development machine for building Poor Man’s Studio from source.

Poor Man’s Studio is a C++ / JUCE / CMake application. The recommended Windows build path uses Microsoft Visual Studio with the MSVC compiler.

1. Required Tools

Install these tools before building the project.

Tool	Required?	Purpose
Windows 10 or Windows 11	Yes	Target development platform
Visual Studio Community, Professional, or Enterprise	Yes	C++ compiler, debugger, IDE, and MSBuild
Desktop development with C++ workload	Yes	Installs MSVC compiler, Windows SDK, and C++ build tools
CMake	Yes	Generates Visual Studio build files from CMakeLists.txt
Git for Windows	Recommended	Cloning, version control, tags, and source management
JUCE source tree	Yes	Audio/application framework used by the app
VST3 plugins	Optional for app launch, needed for VST testing	Used to test VST3 instrument/effect scanning and hosting
CLAP plugins	Optional for app launch, needed for CLAP testing	Used to test CLAP instrument/effect scanning, direct preview, and hosting
Current tested tool versions

2. Recommended Folder Layout

Use a short path with no special characters.

Recommended:

C:\Dev\PoorMansStudio
C:\Dev\JUCE
C:\Dev\VST3SDK

Avoid paths such as:

C:\Users\Your Name\Desktop\New Folder\PoorMansStudio Build Test Copy Final

Long paths and spaces usually work, but they make command-line build troubleshooting harder.

3. Install Visual Studio

Install Visual Studio Community or another Visual Studio edition that includes C++ desktop development.

During installation, select:

Desktop development with C++

Recommended optional components:

MSVC v143 or newer C++ build tools
Windows 10/11 SDK
C++ CMake tools for Windows
MSBuild

After installation, verify that Visual Studio can build C++ projects.

Open:

Start Menu → Visual Studio Installer → Modify

Confirm that Desktop development with C++ is checked.

4. Install CMake

Install CMake. Shawn's current local setup uses CMake 4.4.0.

During setup, choose one of these PATH options:

Add CMake to the system PATH for all users

or:

Add CMake to the PATH for the current user

After installation, open a new Command Prompt and run:

cmake --version

Expected result:

cmake version 4.4.0

A newer stable CMake may also work, but when comparing build logs, treat 4.4.0 as the current local reference.

If Windows says cmake is not recognized, close and reopen Command Prompt. If it still fails, reinstall CMake and enable the PATH option.

5. Install Git for Windows

Install Git for Windows if you plan to clone the repo, commit changes, push tags, or use GitHub.

Recommended setup choices:

Installer Page	Recommended Choice
Default editor	Any editor you are comfortable with
Initial branch name	Git default or project standard
PATH environment	Git from the command line and third-party software
HTTPS transport	OpenSSL
Line endings	Checkout Windows-style, commit Unix-style
Terminal emulator	MinTTY or Windows console, either is okay

Verify installation:

git --version
6. Get JUCE

Poor Man’s Studio uses JUCE as its application/audio framework. Shawn's current local setup uses JUCE 8.0.14.

Recommended folder:

C:\Dev\JUCE

If using Git:

cd C:\Dev
git clone https://github.com/juce-framework/JUCE.git JUCE

If using a ZIP download:

Download JUCE from the official JUCE/GitHub release/source location.
Extract it to:
C:\Dev\JUCE
Confirm this file exists:
C:\Dev\JUCE\CMakeLists.txt

For the current reference setup, confirm you are using JUCE 8.0.14.

If that file is missing, the folder is probably nested one level too deep, such as:

C:\Dev\JUCE\JUCE-main\CMakeLists.txt

In that case, either move the inner contents up or point CMake to the inner folder.

7. Get the Poor Man’s Studio Source

If cloning from GitHub:

cd C:\Dev
git clone <repo-url> PoorMansStudio
cd PoorMansStudio

If using a release ZIP:

Extract the source ZIP to:
C:\Dev\PoorMansStudio
Confirm these files/folders exist:
CMakeLists.txt
src
workspace
README.md

Do not build directly inside the ZIP preview window. Extract the ZIP first.

8. Configure the Build with CMake

From a regular Command Prompt:

cd C:\Dev\PoorMansStudio
cmake -S . -B builds -G "Visual Studio 17 2022" -A x64

If using a newer Visual Studio generator, adjust the generator name as needed. With newer Visual Studio/MSVC installations, it is often easiest to let CMake auto-detect the generator.

For example:

cmake -S . -B builds -A x64

CMake should detect Visual Studio and generate the build files under:

C:\Dev\PoorMansStudio\builds
9. Build from the Command Line

Build the Debug configuration:

cmake --build builds --config Debug

Build the Release configuration:

cmake --build builds --config Release

Debug builds are useful for troubleshooting.

Release builds are better for normal testing.

10. Build from Visual Studio

After configuring with CMake, open the generated solution or folder in Visual Studio.

Common options:

Open Visual Studio → Open a local folder → C:\Dev\PoorMansStudio

or open the generated solution if present under:

C:\Dev\PoorMansStudio\builds

In Visual Studio:

Set configuration to Debug or Release.
Set platform to x64.
Build the project.
Run the Poor Man’s Studio executable.
11. Locate the Built Application

The exact output path can vary by generator/configuration, but common locations include:

builds\Debug
builds\Release
builds\PoorMansStudio_artefacts

Search for:

dir /s /b PoorMansStudio.exe

Run the executable from its output folder or through Visual Studio.

12. Workspace Folder Expectations

The app expects a workspace-style folder layout for runtime files.

Important folders:

Folder	Purpose
workspace/docs	User guide and setup/build guide
workspace/vst3	VST-related runtime files, notes, and shared plugin test sample
workspace/vst3/test	Shared plugin test WAV used by VST3/CLAP Test Effect and related plugin test workflows
workspace/vst3/snapshots	VST effect snapshot files
workspace/temp	Temporary runtime work
workspace/projects	Saved projects
workspace/settings	Local runtime settings

Do not commit local runtime junk unless intentionally part of the source package.

14. VST3 and CLAP Plugin Setup for Testing

Poor Man’s Studio can scan VST3 and CLAP instruments and effects.

Common Windows VST3 locations:

C:\Program Files\Common Files\VST3
C:\Program Files\VST3

Common Windows CLAP locations:

C:\Program Files\Common Files\CLAP

Typical VST3 workflow:

Install one or more VST3 plugins.
Open Poor Man’s Studio.
Use the VST3 Plugin Manager or VST3 scan controls.
Scan the folder containing .vst3 plugins.
Confirm plugins appear as instruments or effects.
Assign an instrument to a MIDI track or assign effects to Effect Slot 1/2.
Use preview/export to test offline processing.

Typical CLAP workflow:

Install one or more CLAP plugins.
Open Poor Man’s Studio.
Use the CLAP Plugin Manager or CLAP scan controls.
Scan the folder containing .clap plugins.
Confirm plugins appear as instruments or effects.
Assign a CLAP Instrument to a MIDI track or assign CLAP Effect entries to Effect Slot 1/2.
Use Preview Track or Preview Project to test direct CLAP preview when safe; mixed or unsupported projects fall back to the rendered preview path.
Open CLAP Settings to check Plugin Graphics Adapter, Max Open CLAP Windows, Safe CLAP Plugin UI Mode, and Refresh Adapter List.

15. Testing VST Instruments

Basic instrument test:

Create or select a MIDI/instrument track.
Choose a scanned VST3 instrument.
Open the instrument editor.
Add MIDI notes in Piano Roll.
Preview playback.
Render/export to confirm the instrument is included.

The VST instrument title bar should include track context so it is clear which track owns the editor.

16. Testing VST Effects

Basic effect test:

Select a track.
Assign a VST effect in VST Effect 1.
Tick Enable for Slot 1.
Click Open Slot 1.
Adjust plugin settings.
Click Apply Changes in the plugin editor.
Click Apply Track Settings in the main window.
Preview or export.

For two effects:

Assign Slot 1.
Assign Slot 2.
Enable both slots.
Preview/export.

Processing order:

source track → VST Effect 1 → VST Effect 2 → preview/export
17. AudioClip Import and Recording Test

Imported and recorded AudioClip tracks should behave the same for VST effect processing.

Important behavior:

Action	Expected Behavior
Import audio into unsaved project	Audio is staged in the active temp session
Record audio into unsaved project	Recording is staged in the active temp session
Save Project	Staged media is committed into the saved project folder
Discard changes	Active staged media is cleaned up
Preview/export	Enabled VST effects are applied to temporary processed files
Source audio	Remains dry and unchanged
18. Clean Rebuild

If the build becomes inconsistent, remove the build folder and regenerate.

cd C:\Dev\PoorMansStudio
rmdir /s /q builds
cmake -S . -B builds -G "Visual Studio 17 2022" -A x64
cmake --build builds --config Release

Do not delete the src or workspace/docs folders.

19. Common Problems
cmake is not recognized

Cause:

CMake is not on PATH.

Fix:

Reinstall CMake.
Enable the PATH option.
Close and reopen Command Prompt.
Run:
cmake --version
Visual Studio generator not found

Cause:

Visual Studio is missing or C++ tools are not installed.

Fix:

Open Visual Studio Installer.
Modify installation.
Install Desktop development with C++.
Re-run CMake.
Windows SDK errors

Cause:

Windows SDK component is missing.

Fix:

Open Visual Studio Installer.
Modify Visual Studio.
Install a Windows 10 or Windows 11 SDK.
Reconfigure CMake.
Build succeeds but app cannot find the shared plugin test WAV

Check that this file exists:

workspace\vst3\test\pms_vst_effect_test_sample.wav

This file should remain a normal runtime file. It should not be embedded into BinaryData. VST3 and CLAP effect test workflows use this sample as the shared audio source; instrument test workflows render through the current plugin state when they need to reflect instrument tweaks.

VST plugins do not appear

Check:

Plugin is actually VST3.
Plugin folder was scanned.
Plugin is installed for the same architecture as the app, usually x64.
Plugin is not blocked by Windows security.
Try rescanning from the VST Plugin Manager.
Imported audio appears in the wrong place

Expected unsaved-project behavior:

workspace\temp\recordings\rec_...\input\audio\imported

Expected saved-project behavior:

workspace\projects\<project name>\input\audio\imported

If imported audio creates a project folder before saving, that is not expected.

20. Packaging Rules

When creating a source bundle, exclude:

.git
builds
workspace\temp
workspace\projects
workspace\settings
workspace\exports
generated zip files
local IDE cache folders
continuation .md handoff files

Keep:

src
CMakeLists.txt
README.md
workspace\docs
workspace\vst3\README.md
workspace\vst3\test\pms_vst_effect_test_sample.wav

The continuation handoff file should be provided separately, not inside the source ZIP.
